## 🎤 Part 2. 발표 내용 정리: "왜 자바는 바로 실행되지 않는가?"

### 1. 우리가 배웠던 내용 (실행 과정)
* **1차 번역 (`javac`):** 사람이 쓴 소스코드(`.java`)를 JVM이 이해할 수 있는 **바이트코드(`.class`)**로 바꿉니다. (컴퓨터는 아직 못 읽음)
* **2차 번역 (JVM):** JVM이 바이트코드를 읽어서 실제 컴퓨터 부품(CPU)이 알아듣는 **기계어(Native Code)**로 바꿉니다.

> 💡 **비유하자면?**
> * **자바 코드:** 한국어 요리법
> * **바이트코드:** 만국 공용 그림 기호 (어느 나라 셰프든 볼 수 있음)
> * **JVM:** 각 나라의 현지 셰프 (그림 기호를 보고 자기 나라 도구로 요리함)

<br>

### 2. Deep Dive: JIT 컴파일러의 비밀
Java가 실행될 때 JVM은 아래 두 가지 방식을 섞어서 사용합니다.

#### ① 인터프리터 (줄 단위 통역사)
프로그램이 시작되면 JVM은 바이트코드를 한 줄씩 읽어서 바로 실행합니다.
* **장점:** 바로 시작할 수 있습니다.
* **단점:** 똑같은 코드를 만 번 실행해도 만 번 다 한 줄씩 읽어야 해서 전체적으로 느립니다.

#### ② JIT 컴파일러 (암기 천재 통역사)
JVM은 실행 중에 "어? 이 코드는 아까부터 계속 반복되는데?" 싶은 부분(**Hotspot**)을 찾아냅니다. 그리고 그 부분을 통째로 기계어로 번역해서 **메모리에 저장(캐싱)**해버립니다.
* **장점:** 다음에 실행할 때는 번역 과정 없이 메모리에서 바로 꺼내 쓰므로 미친 듯이 빨라집니다. (C언어와 맞먹는 속도가 나옵니다.)
* **단점:** "어떤 코드가 중요한지" 지켜보고 번역하는 시간이 필요합니다.

<br>

### 3. 실무 이슈: "웜업 (Warm-up)"
대기업 기술 블로그에서 자주 다루는 자바 서버의 고질적인 문제와 해결책입니다.

* **Cold Start (문제 상황):** 서버를 새로 띄우면 JIT 컴파일러가 아직 중요한 코드를 암기하지 못한 상태입니다. 이때 갑자기 수만 명의 사용자가 접속하면 서버가 버벅거립니다.
* **Solution (서버 예열):** 서버를 클라이언트에게 오픈하기 전에, **가짜 데이터(Mock)**를 많이 보내서 JVM이 중요한 코드를 미리 암기(JIT 컴파일)하게 만듭니다. (최근에는 자동화된 '라이브러리 웜업'도 활용)

<br>

### 4. 결론
* 자바의 중간 단계(바이트코드)가 있기 때문에 윈도우나 OS에서 코드를 수정할 필요 없이 리눅스 서버에 가동이 가능합니다.
* Java는 프로그램이 돌아가는 상황을 실시간으로 보면서 가장 빠른 길을 찾아냅니다.
* **결론:** 어디서나 실행하고, 실행 중에 최적화를 하기 위해 Java는 이러한 복잡한 방식을 택했습니다.

> ➕ **Plus 지식: 파이썬도 JIT가 있나요?**
> 기본 파이썬에는 없지만, `PyPy` 같은 특별한 구현체를 사용하면 자바처럼 JIT를 써서 속도를 높일 수 있습니다.
